## Objective

Demonstrate session hijacking by extracting a valid session cookie from one
browser and injecting it into another browser to gain unauthorized access
without credentials — understanding how session tokens work and how they
can be abused.

---

## Concept — What is Session Hijacking?

When a user logs into a website, the server creates a **session token** —
a unique identifier stored as a cookie. This token authenticates the user
for subsequent requests without requiring repeated login.

```
User Login → Server creates Session Token
           → Token stored as Cookie in Browser
           → Every request sends this Cookie
           → Server validates Cookie = User authenticated
```

**Session Hijacking** = Stealing this token and using it to impersonate
the victim — bypassing authentication entirely.

## Attack Chain

### Step 1 — Login to Target (Browser A — Firefox)

Logged into Facebook using valid credentials on Firefox.
Session established — cookies generated by Facebook server.

**Key cookies identified:**
```
c_user  → Contains Facebook User ID
xs      → Session secret key
datr    → Browser identifier
fr      → Tracking cookie
locale  → Language preference
```

### Step 2 — Extract Session Cookies

Used **Cookie Editor** browser extension to view and export all
session cookies from the authenticated Firefox session.

Cookies copied included: `c_user`, `xs`, `datr`, `fr`, `locale`,
`presence`, `ps_l`, `ps_n`, `sb`, `wd`

### Step 3 — Fresh Browser (Browser B — Chrome)

Opened Chrome browser — no Facebook login, fresh session.
Navigated to facebook.com — login page appeared.

### Step 4 — Inject Stolen Cookies

Using Cookie Editor extension in Chrome:
- Deleted existing empty cookies
- Injected all copied cookies from Firefox session
- Cookie values matched original authenticated session

### Step 5 — Session Hijack Confirmed

Refreshed Facebook page in Chrome.

**Result:**
```
✅ Automatic login — no password required
✅ Full account access gained
✅ Account: Adeel Tariq Pahore
✅ All features accessible — feed, messages, settings
```

Browser B gained complete access to the Facebook account
without any authentication prompt.

## Security Analysis

### Why Did This Work?

Facebook's session cookie (`xs` + `c_user`) was not bound to:
- Browser fingerprint
- IP address
- Device identifier

The server only validated the cookie value — not WHERE it came from.
This allowed a different browser to present the same token successfully.

### Vulnerable Cookie Properties

| Cookie | Role | Risk if Stolen |
|--------|------|----------------|
| c_user | User ID | Account identification |
| xs | Session secret | Full session access |
| datr | Browser ID | Partial bypass |

---

## Defensive Countermeasures

| Control | How it Helps |
|---------|-------------|
| **HttpOnly Flag** | JavaScript cannot access cookies — prevents XSS-based theft |
| **Secure Flag** | Cookies only sent over HTTPS — prevents network interception |
| **SameSite Flag** | Blocks cross-site cookie sending — prevents CSRF |
| **Session Timeout** | Short expiry — stolen tokens expire quickly |
| **IP Binding** | Session linked to IP — different IP = invalid session |
| **MFA** | Second factor required — cookie alone insufficient |
| **Token Rotation** | New token on each request — old tokens immediately invalid |

---

## Key Learnings

- Session tokens are as sensitive as passwords — must be protected equally
- HttpOnly + Secure + SameSite flags together significantly reduce risk
- MFA does NOT protect against session hijacking after login — only at login
- Short session timeouts are the most practical defense
- Token binding to device/IP fingerprint adds strong protection
